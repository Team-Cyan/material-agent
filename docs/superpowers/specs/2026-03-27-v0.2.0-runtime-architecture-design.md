# v0.2.0 Runtime Architecture Design

## Summary

`v0.2.0` replaces the current pipeline-centered runtime with a session/job runtime that is designed for:

- a future GUI shell alongside the CLI shell
- arbitrary model server adapters without assuming OpenAI-compatible APIs
- resumable file-level progress and stage-level recovery
- cleaner boundaries between orchestration, domain rules, and external integrations

This is a full-architecture migration, not a partial cleanup. The old long-term shape based on `Pipeline + State + clients + io` is retired in favor of:

- `shells`
- `app`
- `domain`
- `ports`
- `adapters`

## Goals

- Make CLI and future GUI thin shells over the same application runtime
- Introduce first-class runtime entities: `Session`, `Job`, `JobFile`, `Artifact`, `Event`
- Keep recovery focused on file progress and key stages, not full event replay
- Support any future model backend through capability ports
- Remove cross-layer coupling between scoring, commentary, persistence, progress, and transport

## Non-Goals

- No GUI implementation in `v0.2.0`
- No attempt to keep the old internal module graph as a permanent compatibility layer
- No event-sourcing architecture
- No requirement to preserve the current SQLite schema

## Runtime Model

### Session

Represents one user-level operation context.

Fields:

- `id`
- `kind`
- `input_root`
- `config_snapshot`
- `status`
- `created_at`
- `finished_at`

Examples:

- one CLI `run`
- one future GUI review workspace session

### Job

Represents an executable task inside a session.

Fields:

- `id`
- `session_id`
- `type`
- `stage`
- `status`
- `summary`
- `started_at`
- `finished_at`

Examples:

- `review_photos`
- `rescore`
- `rewrite_xmp`
- `scan_scenes`

### JobFile

Represents file-level execution state and is the core recovery record.

Fields:

- `id`
- `job_id`
- `file_path`
- `group_id`
- `status`
- `rank`
- `error_code`
- `error_message`
- `score_total`
- `scene`
- `scene_raw`

### Artifact

Represents produced data referenced by jobs and files, without forcing every payload into one row.

Fields:

- `id`
- `job_id`
- `job_file_id`
- `kind`
- `uri`
- `metadata_json`

Examples:

- decoded preview jpeg
- fast screening payload
- full score payload
- commentary payload
- xmp write result
- error snapshot

### Event

Represents timeline entries for UI/debugging, but is not the source of truth for recovery.

Fields:

- `id`
- `session_id`
- `job_id`
- `job_file_id`
- `event_type`
- `payload_json`
- `created_at`

## Recovery Strategy

Recovery is snapshot-first:

- `Job.stage` says where the task is at a coarse level
- `JobFile.status` says where each file is at a fine level
- `Artifact` reuses stable intermediate products when available

Recovery is required for:

- file-level progress
- key stages
- write resume after crash

Recovery is not required for:

- full event replay
- exact in-memory progress state reconstruction
- every transient transport detail

## Model Adapter Strategy

Adapters are defined by capability ports, not by protocol families.

Required capability ports:

- `FastScreeningPort`
- `VisionScoringPort`
- `CommentaryPort`
- `RawDecodePort`
- `MetadataWritePort`
- `SessionRepositoryPort`
- `JobRepositoryPort`
- `ArtifactStorePort`
- `EventSinkPort`

This allows:

- OMLX
- Ollama
- future local servers
- future direct in-process inference backends

without changing domain logic.

## Job Flow: review_photos

`review_photos` is the central workflow.

Stages:

1. `discover`
2. `group`
3. `score`
4. `comment`
5. `write`
6. `finalize`

File states:

- `pending`
- `decoded`
- `screened`
- `scored`
- `commented`
- `written`
- `skipped`
- `error`

Rules:

- group boundaries should be persisted, not recomputed during resume
- commentary should be independently retryable
- writing should be idempotent
- file failure should not collapse the whole job unless configured to fail-fast

## Migration Strategy

This version completes all three phases in one branch, but the work is still performed as separate milestones.

### Phase 1

Introduce runtime schema and application services:

- create session/job schema
- route CLI commands through job services
- keep legacy execution behavior working while runtime records are introduced

### Phase 2

Split orchestration from domain and adapters:

- replace `Pipeline` responsibilities with job executor + review job
- move SQLite logic into a dedicated adapter
- split model clients into transport + contract responsibilities
- keep scoring/commentary/grouping logic in domain modules

### Phase 3

Finish GUI-ready and multi-backend-ready surfaces:

- unify progress into event sinks
- expose app-level query APIs for sessions/jobs/files/artifacts
- remove legacy direct command-to-implementation paths

## Data Compatibility

`v0.2.0` is allowed to upgrade the SQLite schema.

The migration should:

- preserve historical score rows when possible
- rebuild derived runtime rows if needed
- prefer explicit migration code over silent shape changes

Backward compatibility for the internal schema is not required.

## Testing Strategy

Tests should be split by layer:

- `domain`: pure unit tests
- `app`: integration tests with fake ports
- `adapters`: contract and live-integration tests
- `shells`: parser/command routing tests

Required regression focus:

- resume after crash
- job/file stage recovery
- OMLX/Ollama adapter contract behavior
- XMP correctness
- session/job/event persistence

## Risks

- schema migration complexity
- temporary duplication while old and new paths coexist
- behavior drift while moving logic out of `Pipeline`
- performance regressions if artifact/event writing becomes too chatty

## Recommended Delivery Shape

Deliver `v0.2.0` as a sequence of milestone commits on branch `r/0.2.0`, with each milestone maintaining:

- passing tests
- working CLI execution
- real photo regression capability

This is a large migration, but it should never become an untestable big-bang rewrite.
